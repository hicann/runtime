# P0/P1/P2 优先级判定标准

## 优先级定义

### P0 — 立即修复

**判定标准：** 阻塞开发者使用核心功能

| 条件（满足任一即为 P0） |
|------------------------|
| 核心 API 无文档，开发者完全无法使用 |
| 关键步骤缺失，端到端流程走不通 |
| 文档与代码严重不一致，导致错误使用 |
| 必要的基础设施缺失（如 LICENSE） |
| 示例代码有编译错误或与当前 API 不兼容 |
| 断点分析中标记为"阻塞（无法继续）"的问题 |

**整改期望：** 尽快修复

### P1 — 近期改善

**判定标准：** 影响开发者体验但不阻塞核心功能

| 条件（满足任一即为 P1） |
|------------------------|
| API 有基本文档但缺少参数详细说明 |
| 缺少代码示例，需要开发者自行摸索 |
| 错误码未完整枚举 |
| 文档存在但质量 C/D 级 |
| 缺少编程模型的系统性说明 |
| 断点分析中标记为"半阻塞（靠猜测勉强推进）"的问题 |

**整改期望：** 近期改善

### P2 — 中长期优化

**判定标准：** 提升生态质量，不影响基本使用

| 条件（满足任一即为 P2） |
|------------------------|
| 缺少 FAQ 或常见问题指南 |
| 缺少横向对比或迁移指南 |
| 文档格式/风格不统一 |
| 缺少架构概览图 |
| 社区贡献流程不够完善 |
| 断点分析中标记为"不阻塞"的体验问题 |

**整改期望：** 中长期优化

## 优先级调整规则

1. **断点分析加权**：如果某个问题在断点分析中被实际撞上，优先级上调一级
2. **多次出现加权**：如果同类问题在多个 API/文档中重复出现，优先级上调一级
3. **衔接缺口**：AscendC 与 Runtime 的衔接问题默认 P1，如果断点分析中为阻塞则升 P0

## 来源标注规则

每个 Roadmap 条目须标注问题来源：

| 标注 | 含义 |
|------|------|
| `[断点#N]` | 来自断点分析的第 N 个断点 |
| `[维度X问题#N]` | 来自静态评估维度 X 的第 N 个问题 |
| `[两者]` | 断点分析和静态评估都发现了此问题 |
