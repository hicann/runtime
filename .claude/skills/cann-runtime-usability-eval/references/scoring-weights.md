# 评分权重

## 维度权重分配

| 维度 | 权重 | 说明 |
|------|------|------|
| 维度一：仓库基础设施 | 10% | 基础门槛，是开发者接触仓库的第一印象 |
| 维度二：API 接口可用性 | 30% | 核心功能，直接决定开发者能否使用 |
| 维度三：docs/ 文档质量 | 25% | 开发者学习 API 的主要依赖 |
| 维度四：example/ 示例质量 | 20% | 开发者的实践学习路径 |
| 维度五：开发者体验 | 15% | 综合体感，影响长期使用意愿 |

## 计算方法

### 单维度评分

每个维度评分范围：1-5 分（整数或 0.5 步进）

### 加权综合得分

```
综合得分 = 维度一 × 0.10 + 维度二 × 0.30 + 维度三 × 0.25 + 维度四 × 0.20 + 维度五 × 0.15
```

### 评分等级映射

| 综合得分 | 等级 | 含义 |
|---------|------|------|
| 4.5 - 5.0 | 优秀 | 开发者可流畅使用 |
| 3.5 - 4.4 | 良好 | 基本可用，有改善空间 |
| 2.5 - 3.4 | 及格 | 可以使用但体验不佳 |
| 1.5 - 2.4 | 不及格 | 多处阻塞，需大量改善 |
| 1.0 - 1.4 | 差 | 基本无法使用 |

## 权重设计理由

- **API 可用性权重最高（30%）**：这是 Runtime 仓库的核心价值，
  API 文档不完整意味着开发者无法使用最基本的功能
- **文档质量次高（25%）**：文档是开发者学习的主要路径，
  直接影响上手效率
- **示例质量（20%）**：示例是"看得见的文档"，
  好的示例能弥补文档的不足
- **开发者体验（15%）**：综合因素，影响长期留存
- **基础设施最低（10%）**：虽然重要，但不是使用的核心障碍
