# 维度三：docs/ 文档完整性与质量检查项

## 权重：25%

## 必要文档类型检查

| 文档类型 | 检查内容 | 严重等级 |
|---------|---------|---------|
| API Reference | 每个接口的入参/出参/错误码/使用约束 | 阻塞性 |
| 编程模型文档 | Device-Context-Stream 层级关系和生命周期 | 阻塞性 |
| 快速上手指南 | 从零到跑通第一个算子的完整步骤 | 阻塞性 |
| 架构概览 | Runtime 在 CANN 软件栈中的位置 | 影响体验 |
| FAQ | 典型报错和解决方案 | 影响体验 |
| 迁移指南 | 从 CUDA/ROCm 迁移的对照 | 优化建议 |

## 每篇文档的检查项

### 内容完整性

| 检查项 | 说明 |
|--------|------|
| 标题和目的 | 文档开头说明目的和适用场景 |
| 前置条件 | 阅读前需要什么知识/完成什么步骤 |
| 正文内容 | 核心知识点是否完整覆盖 |
| 代码示例 | 是否有可执行的代码片段 |
| 注意事项 | 是否标注了常见坑点 |
| 交叉引用 | 是否链接到相关文档和示例 |

### 文档质量

| 检查项 | 说明 |
|--------|------|
| 准确性 | 描述与实际代码是否一致 |
| 清晰度 | 表达是否易于理解 |
| 结构性 | 是否有清晰的章节划分 |
| 可执行性 | 步骤是否可以照做 |
| 时效性 | 是否与当前版本代码同步 |

### 文档间一致性

| 检查项 | 说明 |
|--------|------|
| 术语统一 | 同一概念是否使用相同术语 |
| 命名一致 | API 名称、参数名是否一致 |
| 风格统一 | 写作风格、格式是否统一 |
| 无矛盾 | 不同文档对同一事实的描述是否一致 |
| 语言覆盖 | 是否提供中英双语，或仅有单语言版本 |

## 文档质量打分标准

| 等级 | 标准 | 分数映射 |
|------|------|---------|
| **A** | 内容完整、有示例、与代码同步、有交叉引用 | 5 |
| **B** | 内容基本完整，缺示例或部分参数说明不全 | 4 |
| **C** | 框架存在但内容不完整，关键信息缺失 | 3 |
| **D** | 仅有标题或占位内容，无实质信息 | 2 |
| **F** | 完全缺失 | 1 |

## 评分标准

| 分数 | 标准 |
|------|------|
| 5 | 文档体系完整，每篇质量 A/B 级，术语一致 |
| 4 | 核心文档齐全且质量不错，少数文档缺失或质量 C 级 |
| 3 | 有基本文档但覆盖不全，多篇质量 C/D 级 |
| 2 | 文档严重不完整，大量关键文档缺失或仅占位 |
| 1 | 几乎无有效文档 |
